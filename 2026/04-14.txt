1776157978

not just as much as possible, all zig code in a compilation goes into a single unit

1776158068
ifreund!2940d10f8b@user/ifreund
an artifact could be 1 zig compilation unit + N C compilation units though

1776158091
ifreund!2940d10f8b@user/ifreund

1776158158
ifreund!2940d10f8b@user/ifreund
In C, each foo.c file is generally a compilation unit

1776158205
ifreund!2940d10f8b@user/ifreund
post preprocessing, so all the headers included in that C file are included in that compliation unit as well

1776158255
ifreund!2940d10f8b@user/ifreund
which means a lot of duplicated work since headers are usually included in more than one .c file/compilation unit

1776158291
ifreund!2940d10f8b@user/ifreund
this also means that optimizations like inlining don't work as effectively between .c files

1776158329
ifreund!2940d10f8b@user/ifreund
(LTO is an attempt to mitigate that downside, but is too late in the pipeline really)

1776158377
ifreund!2940d10f8b@user/ifreund
in zig there is always exactly one root source file and exactly one compliation unit

1776158405
ifreund!2940d10f8b@user/ifreund
everything imported by that root source file, including zig files and other modules (e.g. the std) is part of the same compilation unit
