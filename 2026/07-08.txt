dutchie: re: cancelation - error.Canceled is not generally something your code would invent out of thin air.  It's a possible error of most I/O functions, and should generally be return up to the caller.  This is why you tend to see/write a lot of code like: [...] catch |err| switch (err) { error.Foo => ..., error.Bar => ..., error.Canceled => |e| return e }

1783517910
britney!~britney@vps-eb8aba06.vps.ovh.ca

1783518021
blblack!~bblack@wikimedia/bblack-WMF
as the application coder using std.Io (as opposed to implementing std.Io itself), generally you cancel things via Future.cancel() or Group.cancel() or Select.cancel(), and then just about anywhere any other I/O error could happen, you expect the possibility of receiving an error.Canceled and you return it upwards.

1783518068
blblack!~bblack@wikimedia/bblack-WMF
(but there's also fooUncancelable() variants of some I/O calls, and you can also temporarily block cancelation around a critical section with .swapCancelProtection())

1783518110
blblack!~bblack@wikimedia/bblack-WMF
there's also io.recancel() as a way of slightly-deferring the cancel request until the next cancellation point (generally those are I/O calls)

1783518155
blblack!~bblack@wikimedia/bblack-WMF
so you could do: switch(err) { ..., error.Canceled => io.recancel(), }

1783518187
blblack!~bblack@wikimedia/bblack-WMF
(which avoids canceling the task immediately, but causes the error.Canceled to pop up again on the *next* I/O call that can do so)
