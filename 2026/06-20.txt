torque!~tachyon@user/torque
using `struct` is less descriptive (arguably, completely semantically incorrect). The reason it uses @ptrCast is because I was lazily using translate-c to translate the header, but translate-c itself outputs `opaque` declarations for the underlying C structs 

1781974078
torque!~tachyon@user/torque
thus the ptrcasts are between two opaque types and the reason they're there is because `translate-c` will not generate method-like functions inside the opaque object itself. Hand-rolled or modified declarations could get rid of the ptrcasting

1781974475
torque!~tachyon@user/torque
basically, `opaque` prevents the caller from creating the object themselves, whereas a struct (even one with no fields) could be instantiated by the caller. for this style of C api, the user can only ever hold pointers, (a forward declaration is fundamentally uninstantiable) so allowing the zig side to create (incorrect) objects would allow misuse of the API

1781974493
torque!~tachyon@user/torque
which would make it no longer type safe (i.e. there would be a class of error the compiler could not protect you against)
